AI HVAC Agent

An AI-powered customer-service agent for HVAC companies. Customers describe a heating or cooling problem in a chat or over the phone, the AI extracts a structured service request, and admins manage the queue from a dashboard.

Overview #

Customers talk to the agent in plain language. A deterministic triage engine runs a smart, comprehensive ServiceTitan-style intake — a safety screen first (gas/CO/burning/flooding escalate), then qualifying questions (system fully down vs partly working, how long), then a small required gate (issue, urgency, address, and a contact phone), then skippable enrichment (system type, equipment age/brand, property type, owner/renter, warranty, access notes, preferred arrival window, lead source, …). It asks one question at a time with quick-reply chips, so the common path costs 0 tokens, and turns the conversation into a structured service request that staff triage and assign from an admin dashboard. A jobType work classification is derived from the symptom.

Telephone agent (voice)

The same intake agent also answers phone calls via Twilio — call the live demo at (231) 559-9669 to hear it. It is a voice persona, not a second bot — it runs the exact same deterministic router → slot-extraction → state-machine core as the web chat. Speech is captured with Twilio's built-in recognition (<Gather input="speech">) and replies are spoken back with an Amazon Polly neural voice (default Polly.Joanna-Neural, override via TWILIO_VOICE). The call hangs up on a terminal state (request submitted or escalated to a human), and gas/CO/fire/flood emergencies still escalate deterministically. The voice channel is optional — it turns on when TWILIO_ACCOUNT_SID and TWILIO_AUTH_TOKEN are set; without them the app is web-chat-only.

Web chat and phone calls converge on one shared core — a deterministic router, slot extraction, and a session state machine — so both channels behave identically. Chat responses stream token-by-token via the Vercel AI SDK's streamText(); the phone agent captures speech with Twilio and speaks replies back with an Amazon Polly neural voice. Once a conversation grows past the 10-message sliding window, a background after() task folds older turns into a rolling running_summary that is prepended to the model context, keeping long calls and chats coherent at flat per-turn cost. After the stream completes, extraction runs in the onFinish callback using generateObject() against a Zod schema. Once the required fields are present and the customer confirms, a service_request is created and surfaces in the admin dashboard.

Quick Start #

Prerequisites

  • Node.js 20+
  • A Neon PostgreSQL database (free tier works)
  • A DashScope API key from Alibaba Cloud (for Qwen)

Run locally

cd ai-hvac-agent
npm install
npm run db:migrate    # create tables
npm run db:seed       # seed demo data
npm run dev           # start at http://localhost:3000

Example Workflow #

Here is a concrete walkthrough. The customer reports a problem; the AI collects what it needs, confirms, and submits. After each step, the panel below shows what the system does behind the scenes.

Customer
My air conditioner is blowing warm air and making a loud rattling noise.
Assistant
I'm sorry to hear that — let's get this sorted. To gauge urgency: how hot is it where you are right now, is anyone vulnerable in the home (young children or elderly), and is the rattling constant or does it come and go?
Behind the scenes: A customer_session already exists (created on page load, status chatting). The user message is sanitized by the guardrails, saved to the messages table, then streamed to Qwen via streamText(). The assistant reply streams back token-by-token.
Customer
It's very hot here, around 95 degrees, and my elderly mother lives with me. The rattling is constant. This is urgent.
Assistant
Thanks — given the heat and a vulnerable household member, I'm flagging this as HIGH urgency. What is the service address so we can dispatch a technician?
Behind the scenes: After the stream completes, the onFinish callback runs background extraction. The model now infers urgency: high from the heat plus a vulnerable occupant. Issue type resolves toward cooling_not_working. The required fields (issue type, urgency, address) are not all present yet, so the session stays in the extracting phase.
Customer
My name is Jane Doe, phone is 555-123-9876, and the address is 742 Evergreen Terrace, Springfield, IL 62704.
Assistant
Got it. Here's what I have:
• Issue: AC blowing warm air, constant rattling
• Urgency: High
• Address: 742 Evergreen Terrace, Springfield, IL 62704
Can you confirm your phone number is correct and let me know your preferred time?
Behind the scenes: Extraction now has all three required fields, so isExtractionComplete() returns true. The extracted JSON is written to customer_sessions.metadata and the UI shows an extraction summary card with progress pills.
Customer
Yes, 555-123-9876 is correct. Tomorrow morning works best. Please go ahead and submit it.
Assistant
Your request is submitted. A technician will reach out within 2 hours to confirm a tomorrow-morning visit. Submitted
Behind the scenes: The customer confirms in the review dialog. The session moves to confirmed then submitted, a service_request is created with a reference number, and the customer's name, phone, email, and address are encrypted at rest (AES-256-GCM) before storage.

The structured output the extractor aims to produce

The extractor targets the Zod schema in src/lib/ai/extraction-schema.ts. For this conversation:

{
  "issueType": "cooling_not_working",
  "urgency": "high",
  "address": "742 Evergreen Terrace, Springfield, IL 62704",
  "customerName": "Jane Doe",
  "customerPhone": "555-123-9876",
  "customerEmail": null,
  "description": "AC blowing warm air with a constant loud rattling noise; home is ~95°F with an elderly resident.",
  "isHvacRelated": true
}

Only three fields are required to complete extraction — issueType, urgency, and address (see REQUIRED_EXTRACTION_FIELDS). The rest (name, phone, email) are captured when offered. customerEmail is null here because the customer never gave one.

Each chat session moves through these states:

chatting → extracting → confirmed → submitted

                ↘ abandoned (timeout)

Deterministic Answers (0-token routing) #

Most customer turns don't need a language model. Before any message reaches Qwen, it passes through a deterministic intent router — a pure function that matches the message against a 65-intent knowledge base and returns a verdict.

What gets answered without an LLM

  • Common questions / FAQ (service area, pricing, hours) → an instant canned answer.
  • Greetings & meta ("hi", "are you a bot?") → handled inline, never shadowing a real issue in the same message.
  • Emergencies (gas smell, carbon monoxide, smoke) → conservative safety/handoff text with a low confidence threshold, so the router fails toward escalation.
  • Slot collection — the "what's your address?" style prompts plus regex extraction of phone, email, and address — all without a model call.

Only genuinely novel or ambiguous input (open-ended problem descriptions, compound messages, account/status lookups, non-Latin text) falls through to the existing Qwen streamText() path. The router is a cost optimizer, not the brain — the LLM fallback is always reachable. A global ROUTER_ENABLED kill-switch disables the whole layer in one flag if a misroute is ever reported.

Why this matters — before / after

The chat endpoint used to make 2 LLM calls per turn (one streamText for the reply, one generateObject for extraction). On a matched deterministic turn that drops to 0 LLM tokens — no reply call and no extraction call. With FAQ and slot-collection turns making up the bulk of a typical conversation, total token cost per conversation falls sharply. Every deterministic turn is logged with its intent id and confidence so the hit-rate is measured, not assumed.

Customer Experience (Stage 1) #

A pass over the chat focused on transparency, trust, and reduced friction — drawn from the patterns leading support bots use (Intercom, Zendesk, Drift, plus NN/G and accessibility guidance). Most of it costs nothing at runtime.

Transparency & trust

  • AI disclosure in the greeting — the first bubble states plainly that it's an AI, that it can make mistakes, and that "Talk to a Human" is one tap away. Zero runtime cost — it's the deterministic greeting.
  • Clearer handoff copy — the chat header carries a subtitle ("AI assistant · a technician follows up within 2 hrs"), and the escalation dialog now sets a concrete expectation plus a tap-to-call action instead of just showing a number.

Guided intake

  • Intake progress stepper — the flat extraction pills are replaced by an explicit "Step X of 3" stepper with a progress bar and per-field check chips, so the 3-field intake (issue, urgency, address) reads as a finish line.
  • Contextual suggested-reply chips — after a bot turn the customer sees 2–3 tappable next steps (issue shortcuts plus one-tap human handoff) instead of having to type, keeping the conversation on rails.

Feedback & accessibility

  • "Was this helpful? 👍/👎" under assistant answers — a tap posts to POST /api/session/feedback and is stored in audit_log, giving a deflection-quality signal with no new tables.
  • Accessibility — the message region is a role="log" with aria-live="polite", animations are gated behind prefers-reduced-motion, and each new assistant message scrolls its top into view (not the container bottom) so users read from the first line.
  • Branded typing indicator — an "HVAC Assistant is typing…" label appears only on the LLM-fallback path, so canned 0-token replies never fake latency.

Continuity, Cost & Ops (Stage 2) #

Stage 2 hardens the conversation against refreshes, caps the cost of every model call, and gives admins a window into how the AI is performing — all on the existing tables, with no schema change.

Conversation resume across refresh

The httpOnly session cookie persists across a page refresh, so the chat rehydrates its transcript and extracted slots from GET /api/session instead of starting over. A customer who accidentally reloads mid-intake keeps their progress — one of the top abandonment causes, removed. Terminal or already-submitted sessions deliberately start fresh.

Token-cost hardening

  • Output cap — chat completions are capped at maxOutputTokens: 350, bounding the cost of any single reply.
  • Sliding window — only the last 10 messages are sent to the model on both the chat and the extraction calls, so cost-per-turn stays flat even in long conversations.

AI Insights dashboard

A new admin view at /admin/insights (backed by GET /api/admin/ai-insights) reports how the assistant is doing, computed entirely from existing tables:

  • Deflection rate (headline) — the share of assistant replies answered with 0 LLM tokens.
  • Funnel — conversations → requests created (with conversion %), escalated, and abandoned.
  • Deterministic-vs-LLM reply counts and total AI tokens used.
  • 👍/👎 feedback tallies from the new helpfulness control.

Admin Dashboard #

Staff sign in at /admin/login and manage everything from four views: Requests, Conversations, AI Insights (see Continuity, Cost & Ops), and Customers / Technicians.

Service Requests

  • Queue of incoming requests with reference #, customer, issue, urgency (color-coded), status, date, and assigned technician.
  • Stats cards (Pending, Assigned Today, In Progress, Completed Today) auto-refresh.
  • Filter by status; open any row for a detail sheet with customer info, issue details, an Intake Details section showing every field captured during the smart intake (system-down status, duration, system type, equipment age/brand, property type, owner/renter, warranty, access notes, vulnerable occupants, preferred window, contact preference, lead source, derived jobType), a technician-assignment dropdown, and the full conversation transcript.

Customers (CRM)

Persistent customer profiles deduplicated from request data — contact info, property details, equipment registry, service-history timeline, notes, and follow-up reminders. Backed by the customers, customer_equipment, customer_notes, follow_ups, and service_history tables.

Technicians

Add, edit, and activate/deactivate technicians. Only active technicians appear in the assignment dropdown on service requests.

Conversations (new)

A robust log of every saved customer conversation — web chats and phone calls, including sessions that never became a service request. Each session carries a channel of web or phone: the list shows a Channel badge and can be filtered by channel. It is searchable, and each entry opens a detail sheet with the full transcript plus any extracted data. For long conversations, the detail sheet also shows the rolling Conversation Summary (the running_summary produced by summary compaction).

Why this matters

The Conversations view reads directly from the customer_sessions + messages tables, so it stays complete even when a chat finishes without producing a service_request. Nothing a customer typed is lost — regardless of how the conversation ended.

Data model

Defined with Drizzle in src/lib/db/schema.ts. Every table carries an organization_id for multi-tenancy. The core tables are summarised below; for the full picture — all 65 tables grouped by domain, with foreign keys and live row counts — see the interactive database schema diagram.

Following ServiceTitan's Customer/Location model as a soft split: the service address lives on the service_requests row (the location), while billing identity lives on the customers row — there is no separate locations table (deferred). The preferred window we capture is appointment intent, not a real calendar booking.

AI layer

  • Triage engine: a deterministic state machine (src/lib/ai/triage.ts) drives the ServiceTitan-style intake — safety screen first (gas/CO/burning/flooding escalate), then qualifying questions, then the required gate (issue, urgency, address, contact phone), then skippable enrichment (system type, equipment age/brand, property type, owner/renter, warranty, access notes, vulnerable occupants, preferred window, contact preference, lead source). It emits one question at a time with quick-reply chips, so the common path is 0-token; a jobType work classification is derived from the symptom. Shared by web chat and the phone agent. A "no power / thermostat blank" turn first offers safe self-checks (batteries, breaker, filter) to avoid a wasted truck roll.
  • Chat: streamText() (Vercel AI SDK) streams the assistant reply with the system prompt and full conversation history.
  • Extraction: generateObject() with the Zod extractionSchema turns the conversation into typed JSON, run in the onFinish callback so it never blocks the streamed response.
  • Provider: Qwen through DashScope's OpenAI-compatible endpoint, wired via createOpenAI() with AI_BASE_URL, AI_API_KEY, and AI_MODEL.
  • State machine: determineNextState() advances the session based on extraction completeness and turn count; terminal states reject further messages.
  • Summary compaction: once a conversation exceeds the 10-message sliding window, a background after() task folds the older turns into a rolling running_summary that is prepended to the model context, so long calls and chats stay coherent at flat per-turn cost. This is why the per-org turn ceiling can default to 40 (admin-tunable). See src/lib/ai/compaction.ts and compact-session.ts.
  • Voice orchestration: the Twilio webhooks (POST /api/voice/incoming greets and creates a phone session; POST /api/voice/gather handles each spoken turn) feed the same router → extraction → state-machine core as the web chat, then render TwiML that speaks the reply back with an Amazon Polly neural voice. Webhooks are signature-validated (X-Twilio-Signature, HMAC-SHA1 keyed by TWILIO_AUTH_TOKEN) and fail closed; the call hangs up on a terminal state.

Security & multi-tenancy

  • PII encryption: Customer name, phone, email, and address are encrypted at rest with AES-256-GCM (the *_encrypted columns on service_requests and customers). The ENCRYPTION_KEY is required to decrypt.
  • Multi-tenancy: A withTenant() helper scopes every query by organization_id, present on every table.
  • Admin auth: JWT sessions signed with AUTH_SECRET via jose; passwords hashed with bcrypt.
  • Input guardrails: sanitizeInput() strips/flags prompt-injection attempts before any message reaches the model; flagged messages are blocked. Mutation endpoints reject malformed JSON with a 400 before any handler logic runs, and validate id params (404/400 for non-UUIDs, 409 for duplicate unique keys).
  • Rate limiting: A per-IP sliding window throttles chat, auth, and admin mutations. The throttle key comes from clientIp(), which prefers the platform-set x-real-ip over the client-spoofable x-forwarded-for — closing a rate-limit/cost-DoS bypass.
  • Money integrity: The billing engine (pay / refund / void / estimate-accept) is atomic and re-confirmed under live concurrent load; the customer pay endpoint pins the charge to amount == balance. Audit-log entries record field names, ids, and enums only — never PII or dollar amounts.
  • Token budget: Each session has a token budget (default 10,000); requests are rejected once it is exhausted, preventing runaway AI cost.

Security posture was re-validated in a three-round adversarial hardening pass (July 2026) covering every admin, public, technician, and customer-portal surface — see docs/SECURITY_AUDIT.md.

Resolved Issues / Notes #

Resolved — structured extraction against DashScope

Structured extraction previously returned a JSON parse error against the Qwen/DashScope endpoint: the AI SDK's generateObject() defaults to OpenAI's strict json_schema mode, which DashScope does not honor, so the structured object failed to parse and a chat could finish without a service_request being created.

This is now fixed. Extraction uses generateText() with tolerant JSON parsing, so structured extraction works reliably against Qwen/DashScope. As a defense-in-depth measure, the Conversations view still reads customer_sessions and messages directly, so every conversation is preserved regardless of extraction outcome.

Resolved — 3-round adversarial hardening (July 2026)

A three-round autonomous adversarial pass (audit → fix → gate → deploy) swept every admin, public, technician, and customer-portal surface and fixed ~55 real defects: the spoofable-x-forwarded-for rate-limit bypass (now clientIp() / x-real-ip), malformed-JSON→400 on mutations, duplicate-key→409 (Drizzle nests the pg error in .cause), audit-log money-value redaction, and a batch of perf, error-boundary, and accessibility fixes. The billing engine was independently re-confirmed atomic under concurrent load. Test suite green (261 files, 3,781 pass, 0 fail). Round-by-round detail lives in docs/SECURITY_AUDIT.md.
