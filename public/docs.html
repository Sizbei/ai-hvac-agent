AI HVAC Agent

An AI-powered customer-service agent for HVAC companies. Customers describe a heating or cooling problem in a chat, the AI extracts a structured service request, and admins manage the queue from a dashboard.

Overview #

Customers talk to the agent in plain language. The AI asks follow-up questions, collects the details that matter (issue type, urgency, address, contact info), and turns the conversation into a structured service request that staff triage and assign from an admin dashboard.

Stack

Next.js 16 (App Router) React 19 Vercel AI SDK Qwen via DashScope Drizzle ORM Neon PostgreSQL Zod Tailwind CSS 4 jose (JWT) AES-256-GCM

Architecture at a glance

  Customer
     │  natural-language chat
     ▼
 ┌─────────────────────┐   streamText()      ┌──────────────────┐
 │  AI Chat (/chat)     │ ─────────────────▶  │  Qwen via        │
 │  Vercel AI SDK       │ ◀─────────────────  │  DashScope       │
 └─────────────────────┘   streamed tokens    └──────────────────┘
     │  onFinish
     ▼
 ┌─────────────────────┐   generateObject()  ┌──────────────────┐
 │  Structured          │ ─── Zod schema ───▶ │  ExtractionResult│
 │  extraction          │                     │  (JSON)          │
 └─────────────────────┘                      └──────────────────┘
     │  customer confirms
     ▼
 ┌─────────────────────┐                      ┌──────────────────┐
 │  service_request     │ ───────────────────▶│  Admin dashboard │
 │  (PII encrypted)     │                      │  (queue / CRM)   │
 └─────────────────────┘                      └──────────────────┘

Chat responses stream token-by-token via the Vercel AI SDK's streamText(). After the stream completes, extraction runs in the onFinish callback using generateObject() against a Zod schema. Once the required fields are present and the customer confirms, a service_request is created and surfaces in the admin dashboard.

Quick Start #

Prerequisites

  • Node.js 20+
  • A Neon PostgreSQL database (free tier works)
  • A DashScope API key from Alibaba Cloud (for Qwen)

Run locally

cd ai-hvac-agent
npm install
npm run db:migrate    # create tables
npm run db:seed       # seed demo data
npm run dev           # start at http://localhost:3000

Environment variables (.env.local)

Set these as variable names only — never commit real values. Generate secrets with openssl rand -hex 32.

VariablePurpose
DATABASE_URLNeon PostgreSQL pooled connection string
AI_BASE_URLDashScope OpenAI-compatible endpoint
AI_API_KEYYour DashScope API key
AI_MODELModel name (default: qwen-plus)
ENCRYPTION_KEY32-byte hex key for AES-256-GCM PII encryption
AUTH_SECRETJWT signing secret for admin auth (min 32 chars)
CRON_SECRETSecret for the session-cleanup cron job
Demo login

The seed script creates a demo admin: admin@demo-hvac.com / admin123. Sign in at /admin/login and change the password after first login.

Example Workflow #

Here is a concrete walkthrough. The customer reports a problem; the AI collects what it needs, confirms, and submits. After each step, the panel below shows what the system does behind the scenes.

Customer
My air conditioner is blowing warm air and making a loud rattling noise.
Assistant
I'm sorry to hear that — let's get this sorted. To gauge urgency: how hot is it where you are right now, is anyone vulnerable in the home (young children or elderly), and is the rattling constant or does it come and go?
Behind the scenes: A customer_session already exists (created on page load, status chatting). The user message is sanitized by the guardrails, saved to the messages table, then streamed to Qwen via streamText(). The assistant reply streams back token-by-token.
Customer
It's very hot here, around 95 degrees, and my elderly mother lives with me. The rattling is constant. This is urgent.
Assistant
Thanks — given the heat and a vulnerable household member, I'm flagging this as HIGH urgency. What is the service address so we can dispatch a technician?
Behind the scenes: After the stream completes, the onFinish callback runs background extraction. The model now infers urgency: high from the heat plus a vulnerable occupant. Issue type resolves toward cooling_not_working. The required fields (issue type, urgency, address) are not all present yet, so the session stays in the extracting phase.
Customer
My name is Jane Doe, phone is 555-123-9876, and the address is 742 Evergreen Terrace, Springfield, IL 62704.
Assistant
Got it. Here's what I have:
• Issue: AC blowing warm air, constant rattling
• Urgency: High
• Address: 742 Evergreen Terrace, Springfield, IL 62704
Can you confirm your phone number is correct and let me know your preferred time?
Behind the scenes: Extraction now has all three required fields, so isExtractionComplete() returns true. The extracted JSON is written to customer_sessions.metadata and the UI shows an extraction summary card with progress pills.
Customer
Yes, 555-123-9876 is correct. Tomorrow morning works best. Please go ahead and submit it.
Assistant
Your request is submitted. A technician will reach out within 2 hours to confirm a tomorrow-morning visit. Submitted
Behind the scenes: The customer confirms in the review dialog. The session moves to confirmed then submitted, a service_request is created with a reference number, and the customer's name, phone, email, and address are encrypted at rest (AES-256-GCM) before storage.

The structured output the extractor aims to produce

The extractor targets the Zod schema in src/lib/ai/extraction-schema.ts. For this conversation:

{
  "issueType": "cooling_not_working",
  "urgency": "high",
  "address": "742 Evergreen Terrace, Springfield, IL 62704",
  "customerName": "Jane Doe",
  "customerPhone": "555-123-9876",
  "customerEmail": null,
  "description": "AC blowing warm air with a constant loud rattling noise; home is ~95°F with an elderly resident.",
  "isHvacRelated": true
}

Only three fields are required to complete extraction — issueType, urgency, and address (see REQUIRED_EXTRACTION_FIELDS). The rest (name, phone, email) are captured when offered. customerEmail is null here because the customer never gave one.

Session state machine

Each chat session moves through these states:

chatting → extracting → confirmed → submitted
                ↘ escalated
                ↘ abandoned (timeout)
StateMeaning
chattingActive conversation in progress
extractingAI is collecting the required structured fields
confirmedCustomer reviewed and confirmed the extraction
submittedService request created and visible to admins
escalatedCustomer requested a human handoff
abandonedSession expired; cleaned up by the daily cron

Admin Dashboard #

Staff sign in at /admin/login and manage everything from the dashboard.

Service Requests

  • Queue of incoming requests with reference #, customer, issue, urgency (color-coded), status, date, and assigned technician.
  • Stats cards (Pending, Assigned Today, In Progress, Completed Today) auto-refresh.
  • Filter by status; open any row for a detail sheet with customer info, issue details, a technician-assignment dropdown, and the full conversation transcript.

Customers (CRM)

Persistent customer profiles deduplicated from request data — contact info, property details, equipment registry, service-history timeline, notes, and follow-up reminders. Backed by the customers, customer_equipment, customer_notes, follow_ups, and service_history tables.

Technicians

Add, edit, and activate/deactivate technicians. Only active technicians appear in the assignment dropdown on service requests.

Conversations (new)

A robust log of every saved customer chat — including sessions that never became a service request. It is searchable and shows the full transcript plus any extracted data.

Why this matters

The Conversations view reads directly from the customer_sessions + messages tables, so it stays complete even when a chat finishes without producing a service_request (see Known Issue). Nothing a customer typed is lost.

Data model

Defined with Drizzle in src/lib/db/schema.ts. Every table carries an organization_id for multi-tenancy.

TablePurpose
organizationsTenant root; every other row is scoped to one
usersAdmin staff and technicians (role, password hash, active flag)
customer_sessionsOne chat session: token, status, token budget/usage, turn count, extracted metadata
messagesEvery chat message (user / assistant / system) with token usage
service_requestsSubmitted request: issue type, urgency, encrypted PII, reference number, assignment
customersCRM profile with encrypted contact info and property details
customer_equipmentInstalled HVAC units (type, make, model, serial, warranty)
customer_notesFree-form notes per customer (general, follow-up, complaint, compliment)
follow_upsScheduled reminders with due dates and status
service_historyCompleted work, parts, cost, and tech notes linked to a request
audit_logAction/entity audit trail with IP address

AI layer

  • Chat: streamText() (Vercel AI SDK) streams the assistant reply with the system prompt and full conversation history.
  • Extraction: generateObject() with the Zod extractionSchema turns the conversation into typed JSON, run in the onFinish callback so it never blocks the streamed response.
  • Provider: Qwen through DashScope's OpenAI-compatible endpoint, wired via createOpenAI() with AI_BASE_URL, AI_API_KEY, and AI_MODEL.
  • State machine: determineNextState() advances the session based on extraction completeness and turn count; terminal states reject further messages.

Security & multi-tenancy

  • PII encryption: Customer name, phone, email, and address are encrypted at rest with AES-256-GCM (the *_encrypted columns on service_requests and customers). The ENCRYPTION_KEY is required to decrypt.
  • Multi-tenancy: A withTenant() helper scopes every query by organization_id, present on every table.
  • Admin auth: JWT sessions signed with AUTH_SECRET via jose; passwords hashed with bcrypt.
  • Input guardrails: sanitizeInput() strips/flags prompt-injection attempts before any message reaches the model; flagged messages are blocked.
  • Rate limiting: A per-IP sliding window throttles the chat endpoint.
  • Token budget: Each session has a token budget (default 10,000); requests are rejected once it is exhausted, preventing runaway AI cost.

Known Issue / Notes #

Note — extraction against DashScope

Structured extraction via generateObject() currently returns a JSON parse error against the Qwen/DashScope endpoint. The AI SDK defaults to OpenAI's strict json_schema mode, which DashScope does not honor, so the structured object can fail to parse.

The practical effect: a session can complete the chat without a service_request being created. This is exactly why the Conversations view reads the customer_sessions and messages tables directly — it stays robust and shows every conversation regardless of whether extraction succeeded.
