<task type="auto">
  <name>Task 1: End-to-end smoke test</name>
  <files>tests/e2e/smoke.test.ts</files>
  <action>
Create `tests/e2e/smoke.test.ts` -- an integration-level smoke test that exercises the complete user flow by calling route handlers directly with mocked dependencies (same pattern as existing tests in `tests/api/admin/`).

The smoke test is a single `describe("E2E Smoke Test: Full Customer-to-Admin Flow")` block with ordered test cases that build on each other using shared state variables.

**Mocking strategy** (follow existing test patterns):
- Mock `@/lib/db` with the vi.hoisted() chainable proxy pattern used in tests/api/admin/requests.test.ts
- Mock `@/lib/session` to control session tokens
- Mock `ai` module (streamText, generateObject) to return canned responses
- Mock `@ai-sdk/openai` to return a mock model
- Mock `@/lib/auth/session` for admin session (getAdminSession)
- Mock `@/lib/crypto` for encrypt/decrypt

**Test cases (in order):**

1. **"POST /api/session creates a new customer session"**
   - Mock db.insert().values().returning() to return a session object with id, status: "chatting"
   - Mock setSessionCookie to capture the token
   - Call the POST handler from session/route.ts
   - Assert: response status 201, body.success === true, body.data.status === "chatting"

2. **"POST /api/chat sends a message and gets AI response"**
   - Mock db.select() to return the session (status: chatting, within token budget)
   - Mock db.select() for message history (empty array)
   - Mock streamText to return a mock stream with toTextStreamResponse()
   - Mock extractServiceRequest to return a complete extraction
   - Call the POST handler from chat/route.ts with { message: "My AC is broken" }
   - Assert: response status 200 (streaming response created)

3. **"POST /api/session/confirm submits a service request"**
   - Mock db.select() to return session with status: "extracting" (confirmable)
   - Mock db.insert() for service_requests to return { id, referenceNumber: "HVAC-TEST1234" }
   - Mock db.update() for session status change
   - Mock db.insert() for audit_log
   - Call the POST handler with valid extraction data: { issueType: "ac_repair", urgency: "high", description: "AC not cooling", address: "123 Main St", customerName: "Test User", customerPhone: "555-0100" }
   - Assert: response status 201, body.data.referenceNumber exists, body.data.status === "submitted"

4. **"GET /api/admin/requests shows the submitted request"**
   - Mock getAdminSession to return { userId: "admin-id", organizationId: DEMO_ORG_ID }
   - Mock getRequests to return a list containing the service request
   - Call the GET handler
   - Assert: response status 200, body.data.requests is an array with length >= 1

5. **"POST /api/admin/requests/[id]/assign assigns a technician"**
   - Mock getAdminSession for auth
   - Mock db queries for: service request exists, technician exists and belongs to org, update succeeds, audit log insert succeeds
   - Call the POST handler with { technicianId: "tech-uuid" }
   - Assert: response status 200, body.success === true

6. **"All error paths return structured API errors (no stack traces)"**
   - Call POST /api/session with a db error (mock db.insert to throw)
   - Assert: response body matches `{ success: false, error: { message: string, code: string } }`
   - Assert: response body does NOT contain "stack", "at ", or "Error:" patterns
   - Call POST /api/chat without a session token
   - Assert: 401 with structured error

Each test calls the actual exported route handler function (imported directly), constructing NextRequest objects with appropriate headers, cookies, and body. This avoids needing a running server.

Use `vi.mocked()` for type-safe mock access. Use `beforeEach` to reset all mocks. Share state between tests via module-level `let` variables where needed (e.g., sessionId flows from test 1 to test 2).
