    - Test 4: GET with valid CRON_SECRET deletes messages belonging to sessions older than 90 days
    - Test 5: GET with valid CRON_SECRET deletes sessions older than 90 days (after their messages are deleted)
    - Test 6: GET returns a summary with counts of expired and purged items
    - Test 7: Sessions in terminal states (submitted, escalated, abandoned, confirmed) are NOT expired, only purged after 90 days
  </behavior>
  <action>
Create `src/app/api/cron/cleanup/route.ts` with a GET handler:

1. **Auth check**: Compare `request.headers.get("authorization")` against `Bearer ${process.env.CRON_SECRET}`. Return `errorResponse("Unauthorized", "UNAUTHORIZED", 401)` if missing or wrong. This protects the endpoint in production (Vercel Cron sends this header automatically).

2. **Expire stale sessions**: Update `customerSessions` where `status IN ('chatting', 'extracting')` AND `updatedAt < NOW() - 24 hours` to set `status = 'abandoned'`. Use drizzle's `sql` template for the date comparison: `sql`${customerSessions.updatedAt} < NOW() - INTERVAL '24 hours'``.

3. **Purge old data (90-day retention)**:
   - First, select session IDs where `createdAt < NOW() - INTERVAL '90 days'`
   - Delete messages where `sessionId IN (...)` those session IDs
   - Delete the sessions themselves
   - Use drizzle's `inArray` for the deletes
   - Handle the case where there are no sessions to purge (skip deletes)

4. **Return summary**: `successResponse({ expiredSessions: N, purgedSessions: N, purgedMessages: N })`

5. **Logging**: Use `logger.info` with the summary counts.

For tests, mock `db` using the same vi.hoisted() + proxy-based chainable mock pattern used in `tests/api/admin/requests.test.ts`. Mock `process.env.CRON_SECRET = "test-secret"`.
